1. Introducción al Paradigma Funcional

¿Qué es la Programación Funcional?

La programación funcional (FP) trata la computación como la evaluación de funciones matemáticas. Se basa en el λ-cálculo de Alonzo Church (1930) y se distingue del paradigma imperativo porque describe qué calcular, no cómo ejecutarlo paso a paso.

AñoHito
1930Alonzo Church formaliza el λ-cálculo
1958John McCarthy crea LISP, primer lenguaje funcional
1987Nace Haskell, lenguaje puramente funcional
2014Java 8 incorpora lambdas, Stream API e interfaces funcionales
2016+Java 16–21 añade records, sealed classes, pattern matching

Paradigma Imperativo vs. Funcional

❌ Imperativo — cómo
List<Integer> pares = new ArrayList<>();
for (int n : numeros) {
    if (n % 2 == 0) {
        pares.add(n); // mutación
    }
}
✅ Funcional — qué
List<Integer> pares = numeros.stream()
    .filter(n -> n % 2 == 0)
    .toList(); // sin mutación

Principios Fundamentales

PrincipioDescripción
Funciones purasMismo input → mismo output, sin efectos secundarios
InmutabilidadLos datos no cambian; se crean nuevas versiones
HOFsLas funciones son valores que se pasan y retornan
ComposiciónCombinar funciones simples para comportamiento complejo
Transparencia referencialUna expresión puede sustituirse por su valor sin alterar el programa

2. Funciones Puras

Definición

Una función pura cumple exactamente dos condiciones:

  1. Determinismo: dado el mismo input, siempre retorna el mismo output.
  2. Sin efectos secundarios: no modifica ningún estado externo.
💡 Transparencia Referencial
Una expresión tiene transparencia referencial si puede reemplazarse por su valor sin cambiar el comportamiento del programa. Por ejemplo: suma(3,4) puede reemplazarse por 7 en cualquier parte del código.

Funciones Puras vs. Impuras

CriterioPuraImpura
Mismo input → mismo output✅❌
Sin efectos secundarios✅❌
Depende de estado externo❌✅
Fácil de testear✅❌
Paralelizable✅❌

Ejemplo en Java

📄 funciones_puras/FuncionesPuras.java
// ✅ PURA: sólo depende de sus argumentos
public static int suma(int a, int b) {
    return a + b;          // siempre el mismo resultado
}

// ✅ PURA sobre colecciones: retorna NUEVA lista sin mutar la original
public static List<Integer> duplicar(List<Integer> lista) {
    return lista.stream()
                .map(x -> x * 2)
                .toList();  // lista inmutable nueva
}

// ❌ IMPURA: modifica estado externo (efecto secundario)
private static int totalAcumulado = 0;
public static int sumaImpura(int a, int b) {
    totalAcumulado += a + b;  // ← efecto secundario
    return a + b;
}

// ❌ IMPURA: efecto secundario de I/O
public static String saludarImpura(String nombre) {
    System.out.println("Hola, " + nombre); // ← efecto secundario
    return "Hola, " + nombre + "!";
}
⚠️ Efectos secundarios comunes a evitar
Modificar variables globales, leer/escribir archivos, hacer llamadas de red, generar números aleatorios, obtener la hora actual, lanzar excepciones no controladas.

3. Inmutabilidad

Teoría

En programación funcional los datos son inmutables: una vez creado un valor, no puede cambiar. En lugar de modificar un objeto, se crea uno nuevo con los cambios deseados.

Herramientas de Inmutabilidad en Java

HerramientaQué hace
finalProhíbe reasignación de la referencia
recordClase de datos inmutable: campos final, sin setters
List.of()Lista no modificable (lanza excepción si se intenta cambiar)
Collections.unmodifiableList()Vista no modificable de una lista existente
StringInmutable por diseño en Java

Ejemplo en Java — Records

📄 inmutabilidad/Inmutabilidad.java
// Record: inmutable por definición — campos final, sin setters
record Punto(double x, double y) {

    // En vez de modificar, retorna un NUEVO Punto
    Punto mover(double dx, double dy) {
        return new Punto(x + dx, y + dy);
    }
}

// Record con validación en el compact constructor
record Precio(double valor, String moneda) {
    Precio {
        if (valor < 0) throw new IllegalArgumentException("Precio negativo");
        moneda = moneda.toUpperCase();  // normaliza al crear
    }

    Precio aplicarDescuento(double porcentaje) {
        return new Precio(valor * (1 - porcentaje / 100), moneda);
    }
}

// ── Demostración de inmutabilidad ─────────────────────────
var p1 = new Punto(0, 0);
var p2 = p1.mover(3, 4);
// p1 sigue siendo Punto[x=0.0, y=0.0] ← sin cambios
// p2 es un NUEVO objeto Punto[x=3.0, y=4.0]

Mutable vs. Inmutable en Colecciones

❌ Mutable
static void agregarImpuesto(
        double[] precios, double tasa) {
    for (int i = 0; i < precios.length; i++) {
        precios[i] *= (1 + tasa); // MUTA
    }
}
✅ Inmutable
static List<Double> agregarImpuesto(
        List<Double> precios, double tasa) {
    return precios.stream()
        .map(p -> p * (1 + tasa))
        .toList(); // nueva lista
}

4. Funciones de Orden Superior

Definición

Una función de orden superior (HOF) es una función que recibe funciones como argumentos y/o retorna funciones como resultado.

Interfaces Funcionales en Java

InterfazFirmaDescripción
Function<T,R>R apply(T t)Transforma T en R
BiFunction<T,U,R>R apply(T t, U u)Transforma T y U en R
UnaryOperator<T>T apply(T t)Caso especial: T → T
Predicate<T>boolean test(T t)Condición sobre T
Consumer<T>void accept(T t)Consume T sin retorno
Supplier<T>T get()Produce T sin input

Lambdas y Referencias a Métodos

// Lambda: implementación anónima de interfaz funcional
Function<Integer, Integer> doble   = x -> x * 2;
Predicate<String>          noVacio = s -> !s.isBlank();

// Method references — cuatro formas:
Function<String, Integer>  parsear    = Integer::parseInt;          // 1. estático
Function<String, String>   mayus      = String::toUpperCase;        // 2. de instancia
Consumer<String>           imprimir   = System.out::println;        // 3. instancia específica
Function<String, StringBuilder> crear = StringBuilder::new;        // 4. constructor

HOF como Estrategia y HOF que Retorna Función

// HOF que recibe función: aplica estrategia de ordenamiento
public static <T> List<T> ordenar(List<T> lista, Comparator<T> criterio) {
    return lista.stream().sorted(criterio).toList();
}
// Uso con distintas estrategias:
ordenar(personas, Comparator.naturalOrder());
ordenar(personas, Comparator.comparingInt(String::length));

// HOF que RETORNA función: fábrica de validadores
public static Predicate<Double> enRango(double min, double max) {
    return valor -> valor >= min && valor <= max;
}
Predicate<Double> precioValido = enRango(0, 10_000);
Predicate<Double> tempValida   = enRango(-273.15, 1_000_000);

5. Map, Filter y Reduce

Las Tres Operaciones Fundamentales

OperaciónDescripciónFirma
mapTransforma cada elementoList<A> → List<B>
filterSelecciona elementos que cumplen una condiciónList<A> → List<A>
reduceCombina todos en un solo valorList<A> → B
ℹ️ Lazy Evaluation en Streams
Las operaciones intermedias (map, filter) son lazy: no se ejecutan hasta que se llama a una operación terminal (toList, reduce, findFirst). Esto permite optimizaciones importantes.

Ejemplo en Java — Stream API

📄 streams/MapFilterReduce.java
record Producto(String nombre, double precio, String categoria) {}

var productos = List.of(
    new Producto("Laptop",  1200, "tecnologia"),
    new Producto("Libro",   25,   "educacion"),
    new Producto("Monitor", 350,  "tecnologia"),
    new Producto("Teclado", 80,   "tecnologia")
);

// map: transforma cada elemento
var nombres = productos.stream()
                       .map(Producto::nombre)
                       .toList();
// ["Laptop", "Libro", "Monitor", "Teclado"]

// filter: selecciona por condición
var techBarata = productos.stream()
                          .filter(p -> p.categoria().equals("tecnologia"))
                          .filter(p -> p.precio() < 500)
                          .toList();
// [Monitor, Teclado]

// reduce: acumula en un valor
double totalTech = productos.stream()
                            .filter(p -> p.categoria().equals("tecnologia"))
                            .mapToDouble(Producto::precio)
                            .sum();
// 1630.0

// Pipeline encadenado: nombres en mayúsculas de tech < $500
var resultado = productos.stream()
                         .filter(p -> p.categoria().equals("tecnologia"))
                         .filter(p -> p.precio() < 500)
                         .map(p -> p.nombre().toUpperCase())
                         .toList();
// ["MONITOR", "TECLADO"]

// Agrupar por categoría con Collectors
Map<String, List<Producto>> porCategoria = productos.stream()
    .collect(Collectors.groupingBy(Producto::categoria));

6. Currying y Aplicación Parcial

Currying

Currying transforma una función de múltiples argumentos en una secuencia de funciones de un argumento:

f(a, b, c)  →  f(a)(b)(c)

Diferencias

CaracterísticaCurryingAplicación Parcial
ArgumentosUno a la vez siempreUno o más a la vez
ResultadoSiempre otra función (hasta el último)Puede ser valor o función
Uso típicoComposición funcionalEspecialización

Ejemplo en Java

📄 currying/Currying.java
// Curried de 2 argumentos: Function<A, Function<B, C>>
Function<Integer, Function<Integer, Integer>> suma = a -> b -> a + b;

// Aplicación total:
suma.apply(3).apply(4);   // → 7

// Aplicación parcial (fijamos el primero):
var suma5  = suma.apply(5);   // Function<Integer, Integer>
suma5.apply(8);               // → 13
suma5.apply(12);              // → 17

// Curried de 3 argumentos
Function<Double, Function<Double, Function<Double, Double>>> volumen =
    largo -> ancho -> alto -> largo * ancho * alto;

var volCaja5x5 = volumen.apply(5.0).apply(5.0);  // espera solo el alto
volCaja5x5.apply(10.0);  // → 250.0

// Utilidad genérica: convertir BiFunction en curried
static <A, B, C> Function<A, Function<B, C>> curry(BiFunction<A, B, C> f) {
    return a -> b -> f.apply(a, b);
}

// Logger especializado con aplicación parcial
BiFunction<String, String, String> log = (nivel, msg) -> "[" + nivel + "] " + msg;
Function<String, String> logInfo  = partial(log, "INFO");
Function<String, String> logError = partial(log, "ERROR");
logInfo.apply("Servidor iniciado");       // [INFO] Servidor iniciado
logError.apply("Archivo no encontrado"); // [ERROR] Archivo no encontrado

7. Composición de Funciones

Teoría

La composición combina dos funciones para producir una nueva: si f: A → B y g: B → C, entonces g ∘ f: A → C donde (g ∘ f)(x) = g(f(x)).

EstiloNotaciónOrdenEn Java
Pipef ▷ gf primero, luego gf.andThen(g)
Composeg ∘ fg envuelve fg.compose(f)

Ejemplo en Java

📄 composicion/Composicion.java
// Funciones atómicas reutilizables
static UnaryOperator<String> limpiar    = String::strip;
static UnaryOperator<String> normalizar = String::toLowerCase;
static UnaryOperator<String> capitalizar = s ->
    Arrays.stream(s.split(" "))
          .map(w -> Character.toUpperCase(w.charAt(0)) + w.substring(1))
          .collect(Collectors.joining(" "));

// Utilidad pipe: encadena de izquierda a derecha
@SafeVarargs
static <T> Function<T, T> pipe(UnaryOperator<T>... funcs) { ... }

// Composición de 3 pasos simples en 1 función compleja
var formatear = pipe(limpiar, normalizar, capitalizar);
formatear.apply("  juan  ");    // "Juan"
formatear.apply("MARÍA JOSÉ!"); // "María José"

// Extender el pipeline con andThen
Function<String, String> saludoFormal =
    ((Function<String, String>) formatear::apply)
        .andThen(n -> "Estimado/a " + n)
        .andThen(n -> n + ",");
saludoFormal.apply("  ana  "); // "Estimada/o Ana,"

// andThen vs compose
Function<Integer, Integer> doble  = x -> x * 2;
Function<Integer, Integer> sumar1 = x -> x + 1;

var pipe = doble.andThen(sumar1);   // sumar1(doble(x)) → pipe(5) = 11
var comp = doble.compose(sumar1);   // doble(sumar1(x)) → comp(5) = 12

8. Recursión

Teoría

La recursión es el mecanismo por el que una función se llama a sí misma. En FP reemplaza a los bucles: en lugar de mutar un contador, se avanza hacia el caso base en cada llamada.

Recursión Normal vs. Recursión de Cola

❌ Recursión normal — pendiente tras llamada
static long factorial(int n) {
    if (n <= 1) return 1L;
    return n * factorial(n - 1);
    // multiplicación PENDIENTE
}
✅ Recursión de cola — nada pendiente
static long factorialTail(int n, long acc) {
    if (n <= 1) return acc;
    return factorialTail(n - 1, n * acc);
    // llamada ES la última operación
}
⚠️ Java y TCO
Java no optimiza tail-calls automáticamente. Para grandes entradas, usar Stream.iterate() o el patrón trampoline.

Ejemplo en Java

📄 recursion/Recursion.java
// Fibonacci con memoización
private static final Map<Integer, Long> memo = new HashMap<>();
static long fibonacci(int n) {
    if (n <= 1) return n;
    return memo.computeIfAbsent(n, k -> fibonacci(k - 1) + fibonacci(k - 2));
}

// Fibonacci funcional con Stream.iterate (sin stack overflow)
static List<Long> fibonacciStream(int cantidad) {
    record Par(long a, long b) {}
    return Stream.iterate(new Par(0L, 1L), p -> new Par(p.b(), p.a() + p.b()))
                 .limit(cantidad)
                 .map(Par::a)
                 .toList();
}

// Quicksort funcional — puro, sin mutación
static List<Integer> quicksort(List<Integer> lista) {
    if (lista.size() <= 1) return lista;
    int pivote = lista.getFirst();
    var menores = lista.stream().skip(1).filter(x -> x <= pivote).toList();
    var mayores = lista.stream().skip(1).filter(x -> x > pivote).toList();
    return Stream.concat(
        Stream.concat(quicksort(menores).stream(), Stream.of(pivote)),
        quicksort(mayores).stream()
    ).toList();
}

9. Functores y Mónadas

Functor

Un functor es un contenedor que puede ser "mapeado" con una función preservando su estructura. La operación clave es map:

// Optional como functor
Optional<Integer> num    = Optional.of(5);
Optional<Integer> doble  = num.map(x -> x * 2);  // Optional[10]
Optional<String>  s      = Optional.<Integer>empty().map(Object::toString);  // empty

Mónada

Una mónada es un functor que además soporta flatMap para evitar el anidamiento de contenedores:

// map puede anidar: Optional<Optional<T>>
Optional<Optional<String>> anidado = Optional.of("hola")
    .map(s -> Optional.of(s.toUpperCase()));  // ❌ Optional[Optional[HOLA]]

// flatMap aplana el resultado: Optional<T>
Optional<String> correcto = Optional.of("hola")
    .flatMap(s -> Optional.of(s.toUpperCase()));  // ✅ Optional[HOLA]

Ejemplo en Java — Maybe y Result

📄 monadas/Maybe.java · monadas/MonadasDemo.java
// Pipeline con Maybe — sin null-checks manuales
Maybe<String> ciudad = buscarUsuario(1)
    .flatMap(MonadasDemo::obtenerDireccion)
    .flatMap(MonadasDemo::obtenerCiudad);
// Si CUALQUIER paso falla → Nothing (sin NPE)

System.out.println(ciudad);                           // Just(Bogotá)
System.out.println(buscarUsuario(2).flatMap(...)
    .getOrElse("Sin ciudad"));                        // Sin ciudad

// Pattern matching con switch (Java 21+)
String msg = switch (ciudad) {
    case Maybe.Just<String> j   -> "Ciudad: " + j.value();
    case Maybe.Nothing<String> n -> "No hay ciudad";
};

// Result — para operaciones que pueden fallar con error descriptivo
Result<Double> ok  = dividir(100, 4).flatMap(MonadasDemo::raizCuadrada);
Result<Double> err = dividir(100, 0).flatMap(MonadasDemo::raizCuadrada);
System.out.println(ok);   // Ok(5.0)
System.out.println(err);  // Err(División por cero)

Las Tres Leyes de las Mónadas

LeyExpresión
Identidad izquierdaof(a).flatMap(f) ≡ f(a)
Identidad derecham.flatMap(of) ≡ m
Asociatividadm.flatMap(f).flatMap(g) ≡ m.flatMap(x -> f(x).flatMap(g))
